Understanding Linux File Permissions

In Linux, everything is considered a file (even hardware devices and directories). To maintain security, Linux uses a rigid permission system to determine who can read, write, or execute those files.

Every file is owned by a specific User and a Group. Permissions are assigned in three distinct brackets:

  • User (u): The owner of the file.
  • Group (g): Other users who are members of the file's group.
  • Others (o): Everyone else.

The Notation System

Permissions are displayed using either Letters (Symbolic notation) or Numbers (Octal notation).

Symbolic (Letters)

  • r = Read (view contents)
  • w = Write (modify contents)
  • x = Execute (run as a program/script)
  • - = No permission

Octal (Numbers)

  • 4 = Read
  • 2 = Write
  • 1 = Execute
  • 0 = No permission

Example: chmod 755 script.sh gives Read/Write/Execute (4+2+1=7) to User, and Read/Execute (4+1=5) to Group and Others.

Common Commands & Usage

BASH 
# View permissions of files in current directory
$ ls -l

# Change file permissions (Give everyone execute rights)
$ chmod +x payload.sh

# Change file permissions absolutely (rwxr-xr-x)
$ chmod 755 payload.sh

# Change ownership of a file to user 'root' and group 'root'
$ chown root:root sensitive_data.txt

# Recursively change permissions for a folder and its contents
$ chmod -R 700 /home/user/.ssh/

SUID & SGID (Security Risks)

During penetration testing, you often look for special permission bits called SUID (Set User ID) and SGID (Set Group ID). When a file has the SUID bit set, it runs with the privileges of the file's OWNER, not the user running it.

  • If a binary owned by `root` has an SUID bit set, any user executing it briefly acts as `root`.
  • Finding SUID files: find / -perm -4000 -type f 2>/dev/null
